01Next.js Fundamentals

What is Next.js?

A React framework by Vercel that provides production primitives: file-system routing, hybrid rendering (SSR/SSG/ISR/CSR/RSC), image and font optimization, code splitting, middleware, server actions, and a build pipeline (Turbopack/Webpack).

Key value propositions

  • Hybrid rendering per route — mix static, dynamic, streamed, and client-rendered content in one app.
  • Zero-config code splitting — each route is its own bundle; client components ship only when needed.
  • Server-first architecture — React Server Components keep data-fetching and secrets on the server while shipping less JS.
  • Built-in optimization — next/image, next/font, next/script, automatic prefetching.
  • Edge and Node runtimes — same code deploys to serverless functions or the edge.

App Router vs Pages Router

Feature Pages Router App Router
Default component Client Server
Data fetching getStaticProps, getServerSideProps fetch() in RSC, async components, server actions
Layouts _app.tsx only Nested layout.tsx per segment
Loading UI Manual loading.tsx + Suspense streaming
Errors _error.tsx error.tsx per segment
Mutations API routes Server Actions or Route Handlers
Streaming Limited First-class via Suspense
Talking point

The App Router shifts the mental model from "render on client, hydrate on load" to "compose server-rendered trees, opt into interactivity". Server Components reduce client bundle and colocate data logic with UI.

02Routing (App Router)

Nested layouts

Layouts wrap their subtree and preserve state on navigation between sibling routes. Layouts receive children and can be async server components. 

// app/dashboard/layout.tsx
export default async function DashboardLayout({ children }: { children: React.ReactNode }) {
  const user = await getUser()
  return (<div><Sidebar user={user} /><main>{children}</main></div>)
}

Dynamic segments

// app/blog/[slug]/page.tsx
type Props = { params: Promise<{ slug: string }> }
export default async function Post({ params }: Props) {
  const { slug } = await params        // Next 15: params is a Promise
  const post = await getPost(slug)
  return <article>{post.title}</article>
}
export async function generateStaticParams() {
  const posts = await getAllPosts()
  return posts.map((p) => ({ slug: p.slug }))
}
Next 15 change

params, searchParams, cookies(), headers(), and draftMode() are now asynchronous. Must be awaited. Common gotcha in interviews and migrations.

Parallel routes

Render multiple pages in the same layout using named slots (@team, @analytics). Great for dashboards with independent loading/error states. 

// app/dashboard/layout.tsx
export default function Layout({ children, team, analytics }: {
  children: React.ReactNode; team: React.ReactNode; analytics: React.ReactNode
}) {
  return <>{children}{team}{analytics}</>
}

Intercepting routes

Show a route "in place" — e.g. open a photo in a modal over the feed on client nav, but as a full page on refresh / direct link. Uses (.), (..), (...) plus a parallel slot.

Route groups

app/
  (marketing)/layout.tsx  page.tsx        // /
  (app)/layout.tsx        dashboard/page.tsx // /dashboard

Navigation APIs

  • <Link> — declarative, prefetches by default in production.
  • useRouter() from next/navigation — push, replace, refresh, back.
  • redirect(), notFound() from next/navigation — throwable helpers.
  • usePathname(), useSearchParams(), useParams() — client hooks.

03Rendering Strategies Hot

The most-asked topic. Be able to draw and compare all five.

CSR

HTML shell + JS bundle; content rendered in browser. Slow first paint, no SEO by default.

SSR

HTML generated per request. Fresh data, slower TTFB. Any dynamic route in App Router.

SSG

HTML produced at build time. Fastest delivery via CDN, but stale until rebuild.

ISR

Static + revalidation window (revalidate: 60) or on-demand.

RSC / PPR

Static shell with dynamic holes streamed in — best of SSG + SSR.

How Next.js infers the mode

  • Only cached fetch(), no dynamic APIs → static.
  • Uses cookies(), headers(), searchParams, or uncached fetch → dynamic (SSR).
  • fetch(url, { next: { revalidate: N } }) or export const revalidate = N → ISR.
  • "use client" at top → client component, hydrated in browser.
  • Force with export const dynamic = 'force-dynamic' | 'force-static'.

Partial Prerendering (PPR)

Static shell prerendered at build; dynamic components inside <Suspense> streamed at request time. Best of SSG + SSR. 

// next.config.js
module.exports = { experimental: { ppr: 'incremental' } }
// app/product/[id]/page.tsx
export const experimental_ppr = true
export default function Page() {
  return (<>
    <ProductHeader />         {/* static */}
    <Suspense fallback={<Skeleton />}>
      <UserRecommendations /> {/* dynamic, streamed */}
    </Suspense>
  </>)
}

04React Server Components

Mental model

Server Components run only on the server. Never ship JS to client, can be async, can access secrets/DB directly. Client Components run in browser with state and effects.

Capability Server Client
Fetch data / DB access ✅ ❌ (via API)
Access secrets / env ✅ ❌
useState, useEffect ❌ ✅
Event handlers ❌ ✅
Browser APIs (window) ❌ ✅
Async component ✅ ❌
Ships JS to browser No (serialized props only) Yes

The "use client" boundary

A file marked "use client" and everything it imports becomes part of the client bundle. Keep it at leaves of the tree. 

// app/product/page.tsx  (Server Component)
import { AddToCartButton } from './add-to-cart' // client leaf
export default async function Page() {
  const product = await db.product.findFirst()
  return (<>
    <h1>{product.name}</h1>
    <AddToCartButton id={product.id} />
  </>)
}
// app/product/add-to-cart.tsx
'use client'
import { useState } from 'react'
export function AddToCartButton({ id }: { id: string }) {
  const [loading, setLoading] = useState(false)
  return <button onClick={() => addToCart(id)}>Add</button>
}

Composition rules

  • You can pass server components as children to client components — output is serialized as props. Keeps dynamic children out of client bundle.
  • You cannot import a server component into a client component. Pass as children or prop.
  • Props crossing the boundary must be serializable (no functions/class instances; Dates and JSX work).
Interview trick

"How do you use a server component inside a client component?" — You can't import directly, but pass as children. Example: client <Tabs> wrapping server-rendered tab contents.

05Data Fetching

Fetching in Server Components

export default async function Page() {
  const [products, promos] = await Promise.all([
    fetch('https://api.example.com/products', {
      next: { revalidate: 3600, tags: ['products'] },
    }).then((r) => r.json()),
    fetch('https://api.example.com/promos', { cache: 'no-store' }).then((r) => r.json()),
  ])
  return <ProductList products={products} promos={promos} />
}

fetch options that matter

Option Effect
cache: 'force-cache' Cached indefinitely (Data Cache). Default in Next 14.
cache: 'no-store' Skip Data Cache; route becomes dynamic.
next.revalidate: N Time-based ISR (seconds).
next.tags: [...] Tag entries for on-demand invalidation with revalidateTag.
Next 15 default

Fetch is no longer cached by default in Next 15. Opt in with {'{ cache: "force-cache" }'} or {'{ next: { revalidate: N } }'}. Same for GET route handlers. Big migration impact.

Parallel vs sequential (waterfalls)

// ❌ Waterfall — 2× round trips
const user = await getUser()
const posts = await getPosts(user.id)
// ✅ Parallel where possible
const [user, global] = await Promise.all([getUser(), getGlobalFeed()])

Preloading pattern

// utils.ts
export const preloadItem = (id: string) => { void getItem(id) }
export const getItem = cache(async (id: string) => db.item.find(id))
// page.tsx
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params
  preloadItem(id)         // kicks off fetch
  return <Item id={id} />
}

React cache() vs fetch dedup

  • Request memoization: identical fetch() calls in same render deduped automatically by React.
  • cache() (from React): dedupes non-fetch async functions (DB calls) during a single request. 
import { cache } from 'react'
export const getUser = cache(async (id: string) => db.user.find(id))

Client-side data fetching

For interactive/per-user/frequently-updating client data, use SWR or TanStack Query. Combine with server-provided initial data for best UX (no waterfall on first paint).

06Caching Deep Dive Must Know

The #1 source of confusion in Next.js interviews. Four independent caches in the App Router.

Layer Where Purpose Invalidate with
Request Memoization Per-request, React Dedupe identical fetches / cache() calls in one render Ends at request end
Data Cache Server, persistent (across users / deploys) Cache fetch() results Time (revalidate), tags (revalidateTag), path (revalidatePath)
Full Route Cache Server, per build Cache rendered RSC payload + HTML of static routes Redeploy, revalidatePath, revalidateTag
Router Cache Client, in-memory (per session) Cache RSC payloads for back/forward + prefetch router.refresh(), server action returns, tab close

Revalidation APIs

import { revalidatePath, revalidateTag } from 'next/cache'
// In a server action or route handler:
await db.post.create({ ... })
revalidateTag('posts')            // invalidate fetches tagged 'posts'
revalidatePath('/blog')           // invalidate the /blog route

On-demand ISR

// app/api/revalidate/route.ts
import { revalidateTag } from 'next/cache'
import { NextResponse } from 'next/server'
export async function POST(req: Request) {
  const secret = req.headers.get('x-secret')
  if (secret !== process.env.REVALIDATE_SECRET)
    return new NextResponse('Unauthorized', { status: 401 })
  const { tag } = await req.json()
  revalidateTag(tag)
  return NextResponse.json({ ok: true })
}
Next 15 change

The Router Cache no longer caches Page components by default (staleTimes.dynamic = 0). Navigations feel fresher but back/forward can trigger a fetch.

Interview whiteboard: request lifecycle

  1. Browser hits URL → CDN check.
  2. CDN miss → server: check Full Route Cache.
  3. Miss → render RSC tree; each fetch() hits Data Cache first.
  4. Data Cache miss → origin call; store in Data Cache (with tags).
  5. Response streamed to browser; Router Cache stores RSC payload for future navigations.

07Server Actions & Mutations

Call a server function from a client component without hand-writing an API endpoint. Great for forms, likes, add-to-cart. 

// app/actions.ts
'use server'
import { revalidatePath } from 'next/cache'
import { z } from 'zod'
const CreatePost = z.object({ title: z.string().min(1).max(100), body: z.string().min(1) })
export async function createPost(formData: FormData) {
  const parsed = CreatePost.safeParse({
    title: formData.get('title'), body: formData.get('body'),
  })
  if (!parsed.success) return { error: parsed.error.flatten() }
  const user = await getSession()
  if (!user) throw new Error('Unauthorized')
  await db.post.create({ data: { ...parsed.data, userId: user.id } })
  revalidatePath('/blog')
  return { ok: true }
}
// app/new/page.tsx
import { createPost } from '@/app/actions'
export default function NewPost() {
  return (<form action={createPost}>
    <input name="title" /><textarea name="body" />
    <button type="submit">Save</button>
  </form>)
}

Progressive enhancement

Because actions attach to the form's action attribute, submissions work without JavaScript — Next.js posts to a hidden endpoint and re-renders. When JS is available it upgrades to fetch + streamed response.

useFormState and useFormStatus

'use client'
import { useFormState, useFormStatus } from 'react-dom'
function Submit() {
  const { pending } = useFormStatus()
  return <button disabled={pending}>{pending ? 'Saving…' : 'Save'}</button>
}
export function PostForm() {
  const [state, action] = useFormState(createPost, { ok: false })
  return (<form action={action}>
    <input name="title" />
    {state?.error && <p>{state.error.formErrors.join(', ')}</p>}
    <Submit />
  </form>)
}
Security

Server actions are POST endpoints callable by anyone. Always re-check auth/authorization inside the action. Never trust client-supplied user IDs.

Optimistic updates

'use client'
import { useOptimistic } from 'react'
export function Likes({ post, likePost }: { post: Post; likePost: (id: string) => Promise<void> }) {
  const [optimistic, add] = useOptimistic(post.likes, (n) => n + 1)
  return (<form action={async () => { add(1); await likePost(post.id) }}>
    <button>❤ {optimistic}</button>
  </form>)
}

08Streaming & Suspense

Next.js streams HTML chunks as they render. Wrap slow components in <Suspense> and the shell is sent immediately. 

import { Suspense } from 'react'
export default function Page() {
  return (<>
    <Header />
    <Suspense fallback={<FeedSkeleton />}><Feed /></Suspense>
    <Suspense fallback={<SidebarSkeleton />}><Sidebar /></Suspense>
  </>)
}

loading.tsx

Dropping a loading.tsx beside a page automatically wraps page.tsx in a Suspense boundary — no code change needed.

Benefits

  • Faster TTFB and FCP.
  • User sees layout + navigation while slow data loads.
  • Independent failure domains via nested error.tsx.

Caveats

  • Setting Cache-Control on streamed responses is tricky — response not fully known when headers are flushed.
  • Some legacy crawlers still don't handle chunked responses well.

09Middleware & Edge Runtime

middleware.ts runs before a request is handled — perfect for auth guards, A/B tests, geo redirects, bot detection, i18n. Executes on the Edge Runtime (V8 isolates) by default. 

// middleware.ts (project root)
import { NextResponse, type NextRequest } from 'next/server'
export function middleware(req: NextRequest) {
  const token = req.cookies.get('token')
  if (!token && req.nextUrl.pathname.startsWith('/dashboard')) {
    const url = req.nextUrl.clone()
    url.pathname = '/login'
    url.searchParams.set('from', req.nextUrl.pathname)
    return NextResponse.redirect(url)
  }
  const res = NextResponse.next()
  res.headers.set('x-request-id', crypto.randomUUID())
  return res
}
export const config = { matcher: ['/dashboard/:path*', '/api/:path*'] }

Edge vs Node runtime

Edge Node.js
Cold start ~0 ms 100–500 ms

Native modules ❌ ✅

Region Closest to user Fixed region(s)

11Route Handlers (API)

// app/api/users/route.ts
import { NextResponse } from 'next/server'
import { z } from 'zod'
const CreateUser = z.object({ email: z.string().email() })
export async function GET() {
  const users = await db.user.findMany()
  return NextResponse.json(users)
}
export async function POST(req: Request) {
  const body = await req.json()
  const parsed = CreateUser.safeParse(body)
  if (!parsed.success) return NextResponse.json({ error: parsed.error }, { status: 400 })
  const user = await db.user.create({ data: parsed.data })
  return NextResponse.json(user, { status: 201 })
}

export const runtime = 'nodejs'

  • Automatic AVIF/WebP, lazy-loading, responsive srcset.
  • Requires width/height or fill to prevent CLS.
  • Configure allowed remote hosts in next.config.js under images.remotePatterns.

Core Web Vitals targets

  • LCP < 2.5 s — largest image/text block visible.
  • INP < 200 ms — interaction to next paint (replaced FID in 2024).
  • CLS < 0.1 — layout shift.

Next.js perf checklist

  • Push data-fetching to server; keep client bundles small.
  • next/image with priority for hero.
  • next/font to eliminate font-related shifts.
  • Stream with Suspense; add loading.tsx.
  • Prefetch links (default), disable for very large trees.
  • Enable PPR for high-traffic pages.
  • Set Cache-Control on route handlers when safe.
  • Enable experimental.optimizePackageImports for heavy libs.

13Authentication

Patterns

  • Session cookie + middleware: middleware validates signed cookie and redirects. Fast, edge-friendly.
  • JWT in HTTP-only cookie: stateless; verify in middleware or server components.
  • Third-party: NextAuth / Auth.js, Clerk, Kinde, WorkOS. Auth.js has first-class App Router support.

Reading session in RSC

import { cookies } from 'next/headers'
import { verifySession } from '@/lib/session'
export default async function Dashboard() {
  const session = await verifySession((await cookies()).get('session')?.value)
  if (!session) return <LoginRedirect />
  return <Welcome name={session.name} />
}
Best practice

Use middleware for coarse redirects (unauthenticated → login), but enforce authorization in your data layer — server components, server actions, route handlers. Middleware alone is not a security boundary because it doesn't protect API routes called directly.

CSRF

Server Actions include a built-in origin check. For custom mutations use SameSite=Lax cookies and verify Origin/Referer.

14State Management

Server state vs client state

  • Server state — RSC + fetch + revalidate, or SWR/TanStack Query for client fetching.
  • UI state — useState, useReducer, or light stores (Zustand, Jotai).
  • URL as state — filters and pagination in searchParams → shareable, bookmarkable.

Server-seeded, client-managed

// Server: seed initial data
const posts = await getPosts()
return <PostsList initial={posts} />
// Client: SWR takes over
'use client'
import useSWR from 'swr'
export function PostsList({ initial }: { initial: Post[] }) {
  const { data } = useSWR('/api/posts', fetcher, { fallbackData: initial })
  return <ul>{data.map((p) => <li key={p.id}>{p.title}</li>)}</ul>
}
Gotcha

A global store defined in a module can bleed across requests on the server. Wrap client stores in a provider that creates the store per mount, or import them only in "use client" files.

16Security

  • Never expose secrets to client. Env vars prefixed NEXT_PUBLIC_ are inlined into the client bundle.
  • Server-only modules. Add import 'server-only' at top of a module to throw at build if imported by a client component. Symmetric: import 'client-only'.
  • CSP. Send strict Content-Security-Policy via middleware, inject a nonce per request.
  • Rate limiting. Middleware + Upstash Redis or similar edge KV.
  • SSRF. Sanitize server-side fetch targets; don't proxy arbitrary user-supplied URLs.
  • Server Actions. Always re-authorize, validate input with Zod. Never trust action names as authorization. 
// lib/db.ts
import 'server-only'
export const db = new Prisma()

17Deployment & Monitoring

Vercel vs self-host

  • Vercel — first-class: ISR, edge, image optimization, analytics, preview deploys.
  • Self-host via next start (Node) or container. Static export via output: 'export'.
  • Docker: output: 'standalone' → minimal ~50 MB runtime folder.
  • Third-party ISR: Netlify supports it; custom Node servers need to proxy revalidation to shared cache.

Observability

  • instrumentation.ts to wire OpenTelemetry, Sentry, Datadog.
  • Structured JSON logs correlated with a request ID set in middleware.
  • RUM via Vercel Speed Insights or a custom Web Vitals reporter.

export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    await import('./sentry.server.config')
  }
}

18System Design with Next.js Senior

Framework for tackling design rounds:

  1. Clarify requirements & scale. Read/write mix, latency budgets, geo, SEO, personalization.
  2. Rendering per surface.
    • Marketing/docs → SSG or ISR.
    • Product pages → ISR with tag revalidation from CMS.
    • Search / logged-in feed → SSR + streaming, or PPR.
    • Dashboards → SSR shell + client SWR.
  3. Data layer. Postgres via Prisma/Drizzle, read replicas, pooling (PgBouncer, Prisma Data Proxy, Neon serverless driver for edge).
  4. Caching strategy. Data Cache tags per entity; CMS webhook → revalidate; CDN cache headers on static assets; client SWR for real-time-ish.
  5. Auth. Cookie session verified in middleware + re-verified in data layer.
  6. Edge vs Node. Personalization/A/B at edge; heavy compute (PDF, images) in Node or a worker service.
  7. Observability & SLO. Error tracking, tracing, Web Vitals, uptime.
  8. Rollout. Preview deploys per PR, feature flags, canary via middleware weight routing.

Sample: "Design a Twitter clone"

  • Home feed: SSR shell + streamed timeline; SWR appends new tweets via polling / WebSocket.
  • Profile: ISR with per-user tag; revalidate on new tweet action.
  • Compose: server action; optimistic via useOptimistic.
  • Auth: cookie session; middleware guards /home.
  • Search: dynamic SSR reading searchParams.
  • Media: S3 via next/image loader.
  • Fan-out on write for followers; store timelines in Redis.

Sample: "Design e-commerce"

  • PDP: PPR — static above the fold; streamed pricing/inventory/recs.
  • Cart: cookie + server action addToCart → revalidateTag('cart').
  • Checkout: full dynamic SSR (session).
  • CMS-driven marketing: ISR + on-demand revalidation on publish.
  • Search: Algolia/Meilisearch behind a route handler; SSR results for SEO.

Sample: "Design a real-time dashboard"

  • Server shell renders layout + user profile from cookie session.
  • Server component streams initial data via Suspense.
  • Client component opens WebSocket / SSE for live updates.
  • Charts use dynamic import with ssr: false to keep server payload small.

19Common Pitfalls

❌ Sequential awaits

Fetch A then B when they could run in parallel. Use Promise.all.

❌ "use client" too high

Marking a layout as client component turns everything into client code. Push it to leaves.

❌ Importing server code into client

Breaks the build. Use server-only + prop drilling / children.

❌ Ignoring the caches

Not tagging fetches → can't invalidate on mutation. Data goes stale.

❌ Auth only in middleware

Middleware isn't a security boundary for direct API calls. Re-check in data layer.

❌ Missing await on Next 15 APIs

params, cookies(), headers() are now Promises.

❌ Global mutable state on server

Bleeds across requests. Use per-request context or React cache().

❌ Missing alt or dimensions

next/image needs width/height (or fill) to avoid CLS.

❌ Env vars leaked to client

NEXT_PUBLIC_* is inlined into JS bundle — never for secrets.

❌ Trusting client-supplied IDs

Always derive user from session server-side.

❌ Storing session in localStorage

Vulnerable to XSS. Use HTTP-only cookies.

❌ Big client bundles

Import icons/libs at the leaf, prefer tree-shakable packages, use optimizePackageImports.

20Rapid-Fire Q&A

Click any question to expand.

Explain the difference between SSR, SSG, ISR, CSR, and RSC.

CSR renders in the browser; slow first paint, poor SEO. SSR renders HTML on every request; fresh but slower TTFB. SSG renders at build time; fastest but stale. ISR = SSG + timed/on-demand rebuilds. RSC is a component-level model: server components produce serialized UI, sent as an "RSC payload," never shipping their JS. Combined with Suspense you get streaming, and with Partial Prerendering a static shell with dynamic holes.

What is the RSC payload?

A serialized representation of the rendered React tree — a mix of HTML markup, references to client components (chunks to load), and their serialized props. It's what Next sends to the browser after navigation instead of a raw HTML re-render, enabling seamless updates without a full-page reload.

How do you invalidate a cached fetch?

Three options: (1) time-based — next.revalidate: N; (2) tag-based — tag the fetch with next.tags: ['posts'] and call revalidateTag('posts') from a server action / route handler after a mutation; (3) path-based — revalidatePath('/blog').

How do you handle authentication?

Sign a session cookie (HTTP-only, SameSite=Lax, Secure). Verify it in middleware for coarse redirects, then re-verify in the data layer (server components, actions, route handlers) — middleware alone isn't a security boundary. For production, Auth.js / Clerk gives you OAuth, session mgmt, CSRF out of the box.

When would you choose Pages Router today?

Rarely for greenfield. Reasons might be: heavy dependency on a library not compatible with RSC yet, an existing large codebase, or a team not ready to learn the new model. Both routers can coexist during migration.

What's the difference between router.push and router.replace?

push adds an entry to the browser history stack; back button returns to the previous URL. replace swaps the current entry so back skips it. Use replace after login/redirects where you don't want the user to bounce back.

Explain generateStaticParams.

Pre-generate dynamic segments at build. Return an array of param objects; Next builds a static page per entry. Combine with dynamicParams = false to 404 anything not listed, or leave it true (default) to lazily generate on first request (ISR).

What is notFound() and where can you call it?

A throwable helper from next/navigation. Throw it in a server component, layout, or route handler to render the nearest not-found.tsx with a 404 status. Not a hook; can't be called in client components.

Why does Next.js prefer cookies over localStorage for sessions?

Cookies are automatically sent on every request, so the server can render personalized UI in RSC. HTTP-only cookies are invisible to JS, mitigating XSS token theft. LocalStorage is client-only and can't be read during SSR/RSC.

How would you implement A/B testing?

Middleware: assign a bucket cookie on first request, then rewrite the URL to /product/(a)/… or /product/(b)/… route group. Because it's an internal rewrite the URL stays clean. Log the exposure event. For very lightweight experiments, use edge config / feature flags read in a server component.

How do you stream data from an LLM/AI provider?

Route handler with a ReadableStream response, or the Vercel AI SDK's streamText. On the client, consume with useChat or manually with fetch + response.body.getReader(). Set runtime = 'edge' for low-latency TTFB.

How do you migrate from Pages Router to App Router?

Incrementally. Both can coexist: add an app/ directory alongside pages/. Migrate route by route, starting with pages that would most benefit from RSC/streaming (heavy data, personalized dashboards). Replace getServerSideProps/getStaticProps with fetch in server components, and API routes with route handlers or server actions.

Why can Server Components not use hooks?

Hooks depend on React's reconciler and a client-side lifecycle (mount/unmount, batched updates). Server Components run once on the server to produce serialized output — there's no lifecycle to hook into. State/effects belong on the client.

What are common causes of hydration mismatch?

Rendering different HTML on server vs client — e.g. reading Date.now(), Math.random(), window, localStorage, user's locale, or feature-flag results only known client-side. Fixes: keep initial render deterministic; move dynamic-only UI behind useEffect or ssr: false dynamic import.

How do you protect against timing attacks in a session check?

Use crypto.timingSafeEqual when comparing tokens/HMACs. Never compare with === which short-circuits and leaks length/prefix info.

What's the difference between revalidatePath and router.refresh()?

revalidatePath is a server call — invalidates Data Cache + Full Route Cache for a path, affects all users. router.refresh() is a client call — re-fetches the current route's RSC payload for this user only.

Explain Turbopack vs Webpack.

Turbopack is Vercel's incremental bundler written in Rust — much faster HMR and cold start than Webpack. It's the default dev bundler from Next 15 (next dev --turbo) and moving toward parity for production builds. Still opt-in for prod as of writing.

How do you set custom HTTP headers?

Three places: (1) next.config.js headers() function for static rules; (2) middleware for per-request logic; (3) inside a route handler by writing to the returned NextResponse.headers.

How does prefetching work?

<Link> in production prefetches the target route's RSC payload when the link enters the viewport (or on hover). Static routes get the full payload; dynamic ones get just the shell. Disable per link with prefetch={false}, or globally via router options.

What is draftMode()?

An App Router utility that bypasses static rendering for the current request — used for CMS preview modes. Toggled by a signed cookie set from a webhook route. Cache is skipped, letting editors see unpublished drafts.
