01Fundamentals & Mental Model

What React actually is

A UI library that turns a description of the UI (a tree of elements) into DOM mutations. Two packages split responsibilities: react defines the component API and reconciliation algorithm; react-dom (or react-native, react-three-fiber, etc.) is the renderer that commits changes to a host environment.

Core principles

  • Declarative — describe the UI as a function of state; React figures out the transitions.
  • Component-based — small, composable units with local state and explicit boundaries.
  • One-way data flow — props flow down; events bubble up through callbacks.
  • Immutability of state — never mutate; return new references so reconciliation can detect changes cheaply.
  • Referential transparency — a component with the same props and state should produce the same output. Purity is what lets React skip renders, replay renders in Strict Mode, and time-slice with Concurrent React.

Virtual DOM vs Real DOM vs Fiber

Layer Represents Purpose
React Element Plain object { type, props, key } Immutable description of UI at a point in time
Fiber Node Mutable work unit Internal data structure representing a component instance + work in progress
Virtual DOM Element tree in memory Enables diffing without touching real DOM
Host DOM Actual browser nodes What the user sees
Interview framing

Don't say "React is fast because of the Virtual DOM". Say: React is fast because it batches updates, diffs in memory (cheap object comparisons), and commits only the minimal set of DOM mutations — plus, since React 18, it can interrupt rendering to keep the main thread responsive.

React 18 / 19 highlights

  • Concurrent rendering — interruptible renders, transitions, automatic batching everywhere.
  • Suspense for data — components can "wait" for async work.
  • React Server Components — components that never ship JS.
  • Actions & useOptimistic / useFormStatus — first-class async form handling (React 19).
  • use() hook — suspend on any promise or context, inline.
  • React Compiler — auto-memoization at build time (React 19+).

02JSX & Elements

What JSX compiles to

JSX is syntactic sugar for React.createElement (or the new jsx() runtime introduced in React 17). It produces plain immutable objects, not DOM nodes. 

// Source
<button className="primary" onClick={onClick}>Save</button>

// Classic runtime (pre-17)
React.createElement('button', { className: 'primary', onClick }, 'Save')

// Automatic runtime (17+)
import { jsx as _jsx } from 'react/jsx-runtime'
_jsx('button', { className: 'primary', onClick, children: 'Save' })

// Element shape
{ type: 'button', props: { className, onClick, children: 'Save' }, key: null, ref: null }

Rules that trip candidates up

  • Return a single root or use a fragment (<>…</>).
  • Lowercase names are DOM elements; capitalized names are components. <button> vs <Button>.
  • class → className, for → htmlFor, camelCased event props (onClick).
  • style takes an object with camelCased CSS properties. Numeric values default to px.
  • Children of type false, null, undefined, true render nothing. 0 renders the string "0" — classic bug: {items.length && <List />} renders 0 when empty.
Falsy rendering gotcha
// Bug: renders "0" when items is empty
{items.length && <List items={items} />}

// Fix: coerce to boolean or ternary
{items.length > 0 && <List items={items} />}
{items.length ? <List items={items} /> : null}

Keys

Keys tell React which element corresponds to which between renders. Without stable keys, React falls back to index-matching, which corrupts local state on reorder / insert.

  • Keys must be stable, unique among siblings, and predictable.
  • Do not use array index unless the list is static (never reordered, never inserted at head).
  • Keys are not passed as a prop. If you need it inside the child, pass it explicitly as id.
  • Changing a key remounts the subtree — a useful technique to reset state (e.g., form reset on route change).

03Components & Props

Function vs class components

Aspect Function Class
State Hooks (useState, useReducer) this.state
Lifecycle useEffect, useLayoutEffect componentDidMount, etc.
Refs useRef, forwardRef createRef
Error boundary ❌ (must be class) ✅ componentDidCatch, getDerivedStateFromError
Concurrent-ready Fully Legacy lifecycles are marked UNSAFE_
Bundle size / perf Smaller closures, no this binding cost Extra prototype overhead

Function components + hooks are the default. Classes remain only for error boundaries — and even then, wrap with a library like react-error-boundary for convenience.

Composition patterns

1. children

function Panel({ title, children }: { title: string; children: React.ReactNode }) {
  return (
    <section>
      <h2>{title}</h2>
      <div>{children}</div>
    </section>
  )
}

2. Slots via named props

<Layout header={<Nav />} sidebar={<Menu />}>
  <Main />
</Layout>

3. Render props / function children

<MousePosition>
  {({ x, y }) => <p>{x}, {y}</p>}
</MousePosition>

Prop rules seniors are expected to know

  • Props are read-only. Never mutate props; treat them as immutable inputs.
  • Prefer discriminated union types over optional props for variant-based components.
  • Avoid prop-drilling more than 2–3 levels — use composition, context, or state libs.
  • Do not spread unknown props onto DOM elements without filtering — leaks non-standard attributes and warnings.
  • Use as prop or polymorphic components with generics for reusable UI primitives. 
// Polymorphic component
type Props<T extends React.ElementType> = {
  as?: T
  children: React.ReactNode
} & React.ComponentPropsWithoutRef<T>

function Box<T extends React.ElementType = 'div'>({ as, ...rest }: Props<T>) {
  const Cmp = as ?? 'div'
  return <Cmp {...rest} />
}

04State & Lifecycle

What is "state"?

Data that changes over time and affects render output. When state updates, React schedules a re-render of the owning component and its descendants (subject to memoization).

Rules of state

  • Never mutate — state.push(x) is a bug; use setState([...state, x]).
  • Batched — multiple setState in the same handler produce one render (React 18: everywhere, including promises / timeouts).
  • Async in effect — reading state right after setting it returns the old value; use the updater form or an effect.
  • Colocate — put state as close to where it's used as possible; lift only when truly shared.
  • Derived state is a smell — compute from existing state during render instead of storing it. 
// ❌ Storing derived state — must sync manually
const [items, setItems] = useState([])
const [count, setCount] = useState(0) // duplicates items.length

// ✅ Derive at render time — always in sync
const [items, setItems] = useState([])
const count = items.length

Updater function pattern

// Race condition — reads possibly stale `count`
setCount(count + 1)
setCount(count + 1) // still +1, not +2

// Correct — updater sees latest queued state
setCount((c) => c + 1)
setCount((c) => c + 1) // +2

Lifting state up

When two siblings need the same state, hoist it to their nearest common ancestor. Downside: parent re-renders when either child's state changes. Mitigate with useMemo, splitting contexts, or component memoization.

Class lifecycle vs hooks (cheat sheet)

Class Hook equivalent
constructor useState(() => initial)
componentDidMount useEffect(fn, [])
componentDidUpdate useEffect(fn, [deps])
componentWillUnmount cleanup returned from useEffect
getDerivedStateFromProps Compute in render + key reset if needed
shouldComponentUpdate React.memo + useMemo/useCallback
getSnapshotBeforeUpdate useLayoutEffect
componentDidCatch Still class-only

05Hooks Deep Dive Hot

The two rules

  1. Call hooks at the top level. Never inside loops, conditions, or nested functions.
  2. Call hooks only from React functions. Function components or other custom hooks.

Why? React identifies hooks by call order. Each component instance maintains a linked list of hook cells on its Fiber node. If order changes across renders, cells shift and state leaks between hooks. 

// Bug — order changes when `open` toggles
if (open) {
  const [x, setX] = useState(0)
}
useEffect(...)

// Fix — always call, guard the work inside
const [x, setX] = useState(0)
useEffect(() => { if (open) { /* ... */ } }, [open])

useState

  • Returns [value, setValue]. setValue identity is stable across renders.
  • Lazy initializer: useState(() => heavy()) runs only on mount.
  • If updater returns the same value (Object.is), React bails out — no re-render, no child re-render.

useReducer — when to use over useState

  • Multiple related pieces of state that update together (form state, wizard).
  • Next state depends on previous state + action metadata.
  • Testability — reducer is a pure function you can unit test.
  • Dispatch identity is stable, so it's safe to omit from effect deps. 
type State = { count: number; step: number }
type Action = { type: 'inc' } | { type: 'setStep'; step: number }
const reducer = (s: State, a: Action): State => {
  switch (a.type) {
    case 'inc': return { ...s, count: s.count + s.step }
    case 'setStep': return { ...s, step: a.step }
  }
}
const [state, dispatch] = useReducer(reducer, { count: 0, step: 1 })

useEffect

Runs after commit, asynchronously. Used for subscriptions, network, external systems. Do NOT use for computing derived data (do it in render) or synchronizing props → state (usually a design smell). 

useEffect(() => {
  const controller = new AbortController()
  fetch(url, { signal: controller.signal })
    .then((r) => r.json())
    .then(setData)
    .catch((e) => { if (e.name !== 'AbortError') setError(e) })
  return () => controller.abort()   // cleanup on unmount / dep change
}, [url])
Strict Mode double-invoke

In development, React 18 mounts → unmounts → mounts every effect once to shake out bugs. Every effect must be resilient to being re-run (idempotent cleanup, abortable requests). This does not happen in production.

useLayoutEffect

Runs synchronously after DOM mutations, before paint. Use for reading layout (offsets, sizes) and synchronously mutating DOM to avoid flicker. Blocks paint — reserve for cases where useEffect causes visible jank.

useMemo & useCallback

  • useMemo(fn, deps) memoizes the return value of fn.
  • useCallback(fn, deps) === useMemo(() => fn, deps).
  • They are not free — you pay for the deps array + comparison. Use only when a real cost or referential-equality dependency exists.
  • React 19's compiler auto-memoizes — you'll write fewer of these by hand.

Legitimate reasons to memoize

  1. Expensive pure computation (sorting, deriving large lists).
  2. Reference stability for a child wrapped in React.memo.
  3. Reference stability for a dep in another hook (effects / other memos).
  4. Preventing recreation of objects passed to context Provider value.

useRef

  • Mutable box ({ current: T }) whose identity survives renders.
  • Updating ref.current does not trigger a render.
  • Use for: DOM references, timer IDs, latest value snapshots, imperative handles.

useContext

Reads the nearest matching Provider. Every consumer re-renders whenever the Provider's value reference changes. Common trap: passing a fresh object literal each render invalidates every consumer. 

// Every render passes a new object → all consumers re-render
<Ctx.Provider value={{ user, setUser }}>

// Memoize the value
const value = useMemo(() => ({ user, setUser }), [user])
<Ctx.Provider value={value}>

06Advanced & Custom Hooks

useTransition & useDeferredValue

Both mark work as non-urgent. React can interrupt them to keep input responsive.

  • useTransition — you control which updates are transitions via startTransition(() => setState(x)).
  • useDeferredValue(value) — returns a lagged copy of value. Great for filtering large lists as user types. 
const [query, setQuery] = useState('')
const deferred = useDeferredValue(query)
const results = useMemo(() => heavyFilter(items, deferred), [items, deferred])

useId

Generates stable, unique IDs that match between server and client renders. Use for label/input associations, ARIA attributes. Never use as a list key.

useSyncExternalStore

The hook for integrating with external stores (Redux, Zustand, browser APIs like matchMedia). Concurrent-safe — guarantees a consistent snapshot during a render. 

const subscribe = (cb: () => void) => {
  const media = window.matchMedia('(prefers-color-scheme: dark)')
  media.addEventListener('change', cb)
  return () => media.removeEventListener('change', cb)
}
const getSnapshot = () => window.matchMedia('(prefers-color-scheme: dark)').matches
const getServerSnapshot = () => false
const isDark = useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot)

useImperativeHandle

Customizes the instance value exposed to ref. Combined with forwardRef (still needed pre-React 19; React 19 allows ref as a regular prop). 

const Input = forwardRef<{ focus: () => void }, Props>((props, ref) => {
  const inner = useRef<HTMLInputElement>(null)
  useImperativeHandle(ref, () => ({ focus: () => inner.current?.focus() }), [])
  return <input ref={inner} {...props} />
})

useDebugValue

Adds a label for a custom hook in React DevTools. Zero runtime cost in prod.

Custom hooks — the mental model

A custom hook is a function starting with use that calls other hooks. It does not share state between components; each call gets its own instance. Think of custom hooks as a way to encapsulate a protocol, not to share state. 

function useDebounce<T>(value: T, delay = 300): T {
  const [debounced, setDebounced] = useState(value)
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay)
    return () => clearTimeout(id)
  }, [value, delay])
  return debounced
}

function useOnClickOutside(ref, handler) {
  useEffect(() => {
    const listener = (e) => {
      if (!ref.current || ref.current.contains(e.target)) return
      handler(e)
    }
    document.addEventListener('mousedown', listener)
    document.addEventListener('touchstart', listener)
    return () => {
      document.removeEventListener('mousedown', listener)
      document.removeEventListener('touchstart', listener)
    }
  }, [ref, handler])
}

function usePrevious<T>(value: T): T | undefined {
  const ref = useRef<T>()
  useEffect(() => { ref.current = value }, [value])
  return ref.current
}
Senior signal

Great candidates spot the difference between composition (hooks + components) and inheritance. They also know when a "custom hook" is really just a utility function that doesn't need to be a hook.

07Fiber & Reconciliation Deep

Why Fiber exists

The legacy stack reconciler (pre-16) walked the tree recursively — uninterruptible. Long trees blocked the main thread. Fiber re-implements reconciliation as a linked-list of work units that can be paused, resumed, aborted, and prioritized.

Fiber node structure (simplified)

interface Fiber {
  type: string | Function                    // 'div' | Component
  key: string | null
  stateNode: HTMLElement | ComponentInstance // link to host or instance
  child: Fiber | null                        // first child
  sibling: Fiber | null                      // next sibling
  return: Fiber | null                       // parent
  pendingProps: any
  memoizedProps: any
  memoizedState: any                         // linked list of hooks!
  alternate: Fiber | null                    // the other tree (current ↔ WIP)
  flags: number                              // effect side-effect bitmask
  lanes: number                              // priority bitmask
}

Double buffering: current & work-in-progress

React maintains two Fiber trees: the current (mounted) tree and the work-in-progress tree. Updates build the WIP tree; on commit, the pointers swap atomically — no partially rendered state is ever visible.

Reconciliation algorithm (diffing)

  • Compare elements at the same position in the tree.
  • Different types → destroy subtree + mount new (state lost).
  • Same type → reuse instance, update props, recurse.
  • For lists → match by key. Same key = same instance regardless of position.

Diffing is O(n) because React only compares same-level, same-type nodes and assumes stable keys.

Lanes — the priority system

Lane Example trigger
SyncLane flushSync, discrete input handlers
InputContinuousLane Continuous events (mousemove, scroll)
DefaultLane Normal updates outside transitions
TransitionLane startTransition, useDeferredValue
IdleLane Off-screen / <Offscreen> content

Higher-priority lanes can interrupt lower ones. When user types while a transition is rendering, React drops the WIP tree and restarts with the newer input.

Hooks storage

Each Fiber's memoizedState holds a linked list of hook records. useState stores { baseState, queue, next }; useEffect stores tag + dep-array + create/destroy functions. Call order matters because React advances a pointer through this list on every render.

08Render & Commit Phases

Two phases, one update

1. Render phase

Pure. Builds the WIP tree. Can be interrupted, restarted, or discarded. Must not have side effects.

2. Commit phase

Synchronous. Applies DOM mutations, runs refs, layout effects, then passive effects. Cannot be interrupted.

Effect ordering on commit

  1. DOM mutations applied.
  2. useLayoutEffect cleanups (from previous render), then setups. Synchronous.
  3. Refs attached / detached.
  4. Browser paints.
  5. useEffect cleanups, then setups. Asynchronous, after paint.

When does a component re-render?

  • Its own state changed (useState, useReducer).
  • A parent re-rendered and it isn't memoized (or a memoized comparison returned false).
  • A context it subscribes to changed.
  • An external store subscribed via useSyncExternalStore emitted a new snapshot.
  • forceUpdate equivalent — useReducer dispatch with a dummy action.

Bail-outs — how React skips work

  • Same state after setState(x) where Object.is(prev, x) → child not re-rendered.
  • React.memo — shallow-equal props → skip render.
  • PureComponent (class) — shallow compare props and state.
  • Same element identity in output — if you pass children from a parent that didn't change, React reuses them. 
// The `children` trick — Nav element created once and reused across App renders
function App() {
  const [count, setCount] = useState(0)
  return <Layout count={count} onInc={() => setCount((c) => c + 1)}><Nav /></Layout>
}
// Layout re-renders on count change; the <Nav /> child element passed in has
// the same identity, so React bails out on Nav.

09Performance Hot

Diagnose before optimizing

  1. React DevTools Profiler — record an interaction, look at flamegraph + "why did this render?".
  2. Chrome Performance panel — find long tasks (> 50ms), scripting vs rendering vs layout.
  3. Lighthouse / Web Vitals — TTFB, LCP, CLS, INP.
  4. Bundle analyzer (source-map-explorer, webpack-bundle-analyzer) — identify heavy deps.

Rendering optimizations

  • Break up components — smaller components = smaller re-render boundaries.
  • Colocate state — move state down to where it's used to avoid re-rendering unrelated siblings.
  • Move state up + pass children — components passed as children won't re-render just because the wrapping stateful parent did.
  • React.memo — for pure leaf components that receive stable props.
  • Custom equality — React.memo(Comp, (prev, next) => ...). Use sparingly.
  • Split contexts — rapidly-changing values (mouse position) in their own context so slow consumers don't re-render.
  • Windowing / virtualization — react-window / react-virtualized / TanStack Virtual for 10k+ lists.

Bundle optimizations

  • Code split with React.lazy + <Suspense>. Split by route, by feature, by heavy widgets.
  • Tree-shake — import from ESM entry, avoid barrel files that pull the world in.
  • Avoid heavy dependencies — moment.js → date-fns/dayjs; lodash → per-function imports; big icon sets → subset.
  • Compress — Brotli/gzip at the edge.
  • Prefetch / preload — <link rel="preload">, route-level prefetch for likely-next screens. 
const HeavyChart = lazy(() => import('./HeavyChart'))
function Dashboard() {
  return (
    <Suspense fallback={<Skeleton />}>
      <HeavyChart />
    </Suspense>
  )
}

Rendering big lists

import { FixedSizeList } from 'react-window'

function Rows({ items }: { items: string[] }) {
  return (
    <FixedSizeList height={600} itemCount={items.length} itemSize={40} width="100%">
      {({ index, style }) => <div style={style}>{items[index]}</div>}
    </FixedSizeList>
  )
}

Concurrent rendering knobs

  • startTransition — mark expensive updates as interruptible.
  • useDeferredValue — lag heavy computations behind user input.
  • <Suspense> — split rendering into streamed chunks.

Web Vitals for React apps

Metric What it measures React levers
LCP Largest content paint SSR/SSG, streaming, image priority
INP Interaction latency (replaces FID) Concurrent transitions, avoid long tasks
CLS Layout shift Reserve space, avoid late-mount content
TTFB Time to first byte Edge SSR, cache HTML, prerender
TBT / Long Tasks Main thread blocking Split bundles, offload to workers, defer
Interview framing

Answer perf questions with a workflow, not a list of tricks: 1) measure, 2) hypothesize, 3) fix the biggest offender, 4) measure again. Interviewers reward candidates who resist the reflex to sprinkle useMemo everywhere.

10Context & State Management

When to use each

Solution Best for Caveat
Local state (useState) Component-scoped UI state Cannot share across siblings without lifting
Lifted state 2–3 related components Prop drilling past that gets painful
Context Truly global, rarely-changing data (theme, user) All consumers re-render on value change
Redux / Zustand / Jotai Cross-app, complex, high-frequency updates Extra concepts, boilerplate
Server state libs (TanStack Query, SWR) Async data (fetching, caching, invalidation) Not for client-only state
URL state Sharable / bookmarkable state (filters, tabs) String-typed; requires (de)serialization

Context pitfalls

  • Every consumer re-renders when value reference changes — even if it reads only one field.
  • Splitting a monolithic context into narrow contexts (auth, theme, session) is often the right refactor.
  • For selectively subscribing to slices of state, prefer useSyncExternalStore-based libs (Zustand, Redux) or use-context-selector.

Redux (Toolkit) — the shortest possible mental model

// slice
const cart = createSlice({
  name: 'cart',
  initialState: { items: [] as Item[] },
  reducers: {
    add: (s, a: PayloadAction<Item>) => { s.items.push(a.payload) },  // Immer inside
    remove: (s, a: PayloadAction<string>) => { s.items = s.items.filter(i => i.id !== a.payload) },
  },
})
// store
export const store = configureStore({ reducer: { cart: cart.reducer } })
// component
const items = useSelector((s: RootState) => s.cart.items)
const dispatch = useDispatch()
dispatch(cart.actions.add(item))

Zustand — minimal alternative

const useCart = create<{
  items: Item[]
  add: (i: Item) => void
}>((set) => ({
  items: [],
  add: (i) => set((s) => ({ items: [...s.items, i] })),
}))
// Fine-grained subscription — component only re-renders when items.length changes
const count = useCart((s) => s.items.length)

Server state: TanStack Query

Server state is different: it can go stale, must be cached, revalidated, and deduped across components. Do not reinvent this with useEffect + useState. 

const { data, isLoading, error } = useQuery({
  queryKey: ['user', id],
  queryFn: () => fetch(`/api/user/${id}`).then((r) => r.json()),
  staleTime: 60_000,
})

11Refs & DOM

What refs are for

  • Accessing DOM nodes (focus, measure, integrate with a non-React library).
  • Storing mutable values that survive renders without triggering renders.
  • Exposing an imperative API from a component (via useImperativeHandle).

Callback refs vs object refs

// Object ref — most common
const ref = useRef<HTMLDivElement>(null)
<div ref={ref} />

// Callback ref — runs on mount/unmount; useful when you need to react to the node
const setNode = useCallback((node: HTMLDivElement | null) => {
  if (node) resizeObserver.observe(node)
}, [])
<div ref={setNode} />

forwardRef (pre-React 19)

const FancyInput = forwardRef<HTMLInputElement, Props>(({ label, ...rest }, ref) => (
  <label>
    {label}
    <input ref={ref} {...rest} />
  </label>
))
React 19

forwardRef is no longer needed. ref is a regular prop on function components. Existing code keeps working; new code can drop forwardRef.

Refs vs state — decision guide

  • Does the UI need to re-render when it changes? → state.
  • Do you need to preserve a value across renders without re-rendering? → ref.
  • Do you need to read DOM measurements? → ref + useLayoutEffect.

12Error Boundaries

What they catch

  • Errors thrown during rendering, in lifecycle methods, and in constructors of the whole tree below.

What they do NOT catch

  • Event handlers (use try/catch).
  • Asynchronous code (use try/catch or promise handlers).
  • Server-side rendering (use framework-specific mechanisms).
  • Errors thrown in the boundary itself.

Implementation

class ErrorBoundary extends React.Component<{ fallback: React.ReactNode }, { error: Error | null }> {
  state = { error: null as Error | null }
  static getDerivedStateFromError(error: Error) { return { error } }
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    reportToSentry(error, info)
  }
  render() {
    return this.state.error ? this.props.fallback : this.props.children
  }
}

<ErrorBoundary fallback={<ErrorFallback />}>
  <Feature />
</ErrorBoundary>

In practice, use react-error-boundary — it exposes a hook-friendly API, reset semantics, and fallback render props.

Pattern

Wrap feature boundaries (route, dashboard tile, modal) rather than the whole app. That way, a bug in one widget doesn't blank out the entire screen.

13Suspense & Concurrent React 18/19

The Suspense contract

A component can suspend by throwing a promise. The nearest <Suspense fallback> catches it, shows the fallback, and re-renders when the promise resolves. Frameworks (Next.js, Remix) and libraries (TanStack Query, Relay) integrate with this contract. 

<Suspense fallback={<Spinner />}>
  <UserProfile id={id} />   {/* may suspend on data */}
</Suspense>

The use() hook (React 19)

import { use } from 'react'
function Comments({ commentsPromise }: { commentsPromise: Promise<Comment[]> }) {
  const comments = use(commentsPromise)  // suspends until resolved
  return <ul>{comments.map((c) => <li key={c.id}>{c.text}</li>)}</ul>
}
// Unlike other hooks, `use` can be called conditionally.

Concurrent features summary

  • Automatic batching — updates in promises, timeouts, native events are all batched into a single render.
  • Transitions — mark low-priority updates so urgent ones (typing) stay responsive.
  • Streaming SSR — the server flushes HTML as parts of the tree become ready.
  • Selective hydration — client hydrates ready subtrees first; interacting with a not-yet-hydrated part prioritizes its hydration.

Concurrent mental model

Renders are now attempted; they can be discarded before commit. Never rely on a render actually happening — do all real work in commit (effects / event handlers). Ensure components are pure so re-attempts are safe.

  • Actions — form actions and useActionState replace many useState + onSubmit patterns.
  • useOptimistic — declarative optimistic UI while an action is pending.
  • useFormStatus — access pending, data, method, action from any child inside a <form>.
  • use() hook — read promises / context inline, can appear in loops/conditions.
  • ref as a prop — no more forwardRef.
  • Document metadata — <title>, <meta>, <link> in any component are hoisted to <head>.
  • Preloading APIs — preload, preinit, prefetchDNS, preconnect from react-dom.
  • React Compiler — opt-in build tool that auto-inserts memoization; fewer manual useMemo/useCallback.
  • Improved hydration errors — single diff-style message instead of a chain of console warnings. 
// React 19 form action + useActionState + useOptimistic
function Likes({ initial }: { initial: number }) {
  const [likes, addLike] = useActionState(async (prev: number) => {
    await api.like()
    return prev + 1
  }, initial)
  const [optimistic, addOptimistic] = useOptimistic(likes, (curr) => curr + 1)
  return (
    <form action={() => { addOptimistic(); addLike() }}>
      <button type="submit">❤ {optimistic}</button>
    </form>
  )
}

15Server Components & Actions RSC

Mental model

React Server Components (RSC) run only on the server. They ship zero JS to the client — only their serialized output. Client Components ("use client") run on both server (for initial HTML) and client (for interactivity).

Server vs Client — capability matrix

Capability Server Client
DB / filesystem access ✅ ❌
Secrets / env ✅ ❌
useState, useEffect ❌ ✅
Event handlers ❌ ✅
Browser APIs ❌ ✅
Async component ✅ ❌
Ships JS to browser No Yes

Composition rules

  • You cannot import a server component into a client component. Pass as children.
  • Props crossing the boundary must be serializable (no functions, no class instances). Dates and JSX work.
  • Server → Client is one-way. Client can't "call back" into the server tree except via server actions. 
// server component
import ClientTabs from './client-tabs'
import ServerContent from './server-content'
export default function Page() {
  return <ClientTabs><ServerContent /></ClientTabs>   // server child passed to client parent
}

Server Actions

// actions.ts
'use server'
export async function createPost(formData: FormData) {
  const title = formData.get('title') as string
  await db.post.create({ title })
  revalidatePath('/blog')
}

// form.tsx (client)
'use client'
import { createPost } from './actions'
export function Form() {
  return (
    <form action={createPost}>
      <input name="title" />
      <button>Publish</button>
    </form>
  )
}
Security

Server Actions are POST endpoints. Treat them like any API: validate inputs (Zod), check authorization, guard against CSRF (framework usually handles this via cookies + origin checks).

16Design Patterns

Compound components

Multiple components that share implicit state via context to give a flexible API. 

const TabsCtx = createContext<{ active: string; set: (id: string) => void } | null>(null)

function Tabs({ defaultValue, children }: { defaultValue: string; children: React.ReactNode }) {
  const [active, setActive] = useState(defaultValue)
  const value = useMemo(() => ({ active, set: setActive }), [active])
  return <TabsCtx.Provider value={value}>{children}</TabsCtx.Provider>
}
Tabs.List = function List({ children }) { return <div role="tablist">{children}</div> }
Tabs.Trigger = function Trigger({ id, children }) {
  const ctx = useContext(TabsCtx)!
  return <button role="tab" aria-selected={ctx.active === id} onClick={() => ctx.set(id)}>{children}</button>
}
Tabs.Panel = function Panel({ id, children }) {
  const ctx = useContext(TabsCtx)!
  return ctx.active === id ? <div role="tabpanel">{children}</div> : null
}

// Usage
<Tabs defaultValue="a">
  <Tabs.List>
    <Tabs.Trigger id="a">One</Tabs.Trigger>
    <Tabs.Trigger id="b">Two</Tabs.Trigger>
  </Tabs.List>
  <Tabs.Panel id="a">One panel</Tabs.Panel>
  <Tabs.Panel id="b">Two panel</Tabs.Panel>
</Tabs>

Render props / function children

Useful for headless components (Radix, Headless UI, TanStack Table). Consumer controls markup; provider owns behavior.

Higher-order components (HOC)

Function that takes a component and returns a wrapped one. Common historically (connect, withRouter). Prefer hooks now; HOCs still work for cross-cutting concerns that can't be a hook (e.g., class-based error boundary wrappers).

Provider pattern

Encapsulate context + state + actions in a single component. Consumers use a matching hook. 

const AuthCtx = createContext<Auth | null>(null)
export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null)
  const value = useMemo(() => ({ user, login: setUser, logout: () => setUser(null) }), [user])
  return <AuthCtx.Provider value={value}>{children}</AuthCtx.Provider>
}
export function useAuth() {
  const ctx = useContext(AuthCtx)
  if (!ctx) throw new Error('useAuth must be inside AuthProvider')
  return ctx
}

State reducer pattern

Consumer passes a reducer override to a component so they can intercept internal state transitions. Popularized by Kent C. Dodds' downshift.

Controlled vs uncontrolled components

  • Controlled — parent owns value via value + onChange. Easier to validate; can lag on rapid input.
  • Uncontrolled — DOM owns value; read via ref or on submit. Cheaper for large forms.
  • Hybrid — defaultValue initializes, then form library manages via refs.

Container / Presentational (legacy, still useful)

Split components into "smart" (data-fetching, stateful) and "dumb" (pure UI). Improves testability. Today, custom hooks usually replace the container.

18Testing

Testing pyramid for a React app

  • Unit — pure functions, reducers, custom hooks (@testing-library/react renderHook).
  • Component / integration — render user-facing components with RTL, mock at the network layer with MSW.
  • End-to-end — Playwright / Cypress. Cover critical flows only.
  • Visual regression — Chromatic / Percy for design systems.

Testing Library principles

  • Query by accessible role and name first, then label, then text. Avoid data-testid.
  • Simulate real user with userEvent (typing, clicking, keyboard).
  • Assert what the user sees, not implementation details (never inspect state or props).

19Accessibility

Non-negotiables

  • Use semantic HTML first: button, a, label, nav, main. Do not build a button out of a div.
  • Every interactive element must be keyboard operable: Enter/Space, arrow keys for composite widgets.
  • Provide visible focus rings. Never outline: none without a replacement.
  • Associate <label> with inputs — htmlFor + id or wrapping.
  • Use ARIA sparingly: only when semantic HTML can't express the pattern (tabs, combobox, tree). Never overuse.

Live regions & announcements

// Announce async status to screen readers
<div role="status" aria-live="polite">{isLoading ? 'Loading…' : ''}</div>

Focus management

  • Return focus to the trigger when closing a modal.
  • Trap focus inside modals (or use <dialog> with showModal()).
  • Skip links for keyboard users to bypass repeated nav.

Tooling

  • eslint-plugin-jsx-a11y — catches common issues at lint time.
  • @axe-core/react — runtime audits in development.
  • Test with screen readers (VoiceOver, NVDA), not just automated tools.

20Security

Cross-site scripting (XSS)

  • React escapes text children by default — this is your first line of defense.
  • dangerouslySetInnerHTML bypasses that. Sanitize server-side or with DOMPurify before using.
  • URLs from user input in href/src: reject javascript: schemes. 
// Safe: React escapes braces
<p>{userInput}</p>

// Dangerous — sanitize first
<p dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(html) }} />

CSRF

  • SameSite cookies (default Lax in modern browsers) mitigate most CSRF for form POSTs.
  • For non-idempotent APIs, require a custom header or CSRF token.
  • Server actions in RSC frameworks bundle CSRF protections by default.

Authentication

  • Store tokens in HttpOnly, Secure, SameSite cookies. Avoid localStorage for auth tokens — vulnerable to XSS.
  • Use short-lived access tokens + refresh tokens with rotation.
  • Never render secrets in client bundles. Anything imported from a "use client" file becomes public.

Content Security Policy

Set a strict CSP. React's inline event handlers do not use inline scripts, so unsafe-inline is not required for React itself. Watch for third-party widgets that inject inline scripts.

Dependencies

  • Run npm audit / pnpm audit, Snyk, or Dependabot on PRs.
  • Pin dependencies. Lock files must be committed.
  • Audit any dep that touches dangerouslySetInnerHTML or eval-like patterns.

function useLocalStorage<T>(key: string, initial: T) {
  const [value, setValue] = useState<T>(() => {
    const raw = localStorage.getItem(key)
    return raw ? (JSON.parse(raw) as T) : initial
  })
  useEffect(() => { localStorage.setItem(key, JSON.stringify(value)) }, [key, value])
  return [value, setValue] as const
}

  • any anywhere — use unknown + narrowing.
  • Overuse of as — real fix is usually a type guard.
  • Optional props masquerading as required — use discriminated unions instead.

22Pitfalls & Bugs

Stale closures in effects

// Bug — reads count from first render forever
useEffect(() => {
  const id = setInterval(() => console.log(count), 1000)
  return () => clearInterval(id)
}, [])  // missing count

// Fixes: include count in deps, OR use ref, OR use updater form of setState

Infinite render loop

// Bug — new object every render → effect runs every render → setState → new object…
useEffect(() => { setData({ ...data, updatedAt: Date.now() }) }, [data])

// Fix: only depend on what actually changed; or use a ref for the "did I run" flag

Missing deps warning — never suppress blindly

The exhaustive-deps rule is right ~95% of the time. If you truly need to skip a dep, use a ref (useLatestRef) rather than // eslint-disable-next-line.

Reading state right after setting

// setCount schedules; the next line still sees the old value
setCount(count + 1)
console.log(count) // old

Fetching in render

Never call fetch() at the top of a component body. Effects, event handlers, or dedicated data libraries only.

Mutating state directly

state.list.push(x)         // Bug — same reference, no re-render
setState({ ...state, list: [...state.list, x] })  // Correct

Overusing useEffect

  • Setting state from props → derive it, or reset with a key.
  • Kicking off derived data → use useMemo.
  • Chained cascading effects → coalesce into one, or use a reducer.
  • Notifying parents → do it in the event handler that caused the change.

Hydration mismatches

Server-rendered HTML must match the client's first render. Common causes: Date.now(), Math.random(), user-locale formatting, or reading window during render. Guard with useSyncExternalStore or defer to useEffect.

Memoizing everything

Memoization has a cost. If children are cheap and props aren't referentially unstable dependencies elsewhere, React.memo can be a pessimization.

Debounced search input with race-condition-safe results 
function SearchBox({ onSearch }: { onSearch: (q: string) => Promise<Result[]> }) {
  const [q, setQ] = useState('')
  const [results, setResults] = useState<Result[]>([])
  const [loading, setLoading] = useState(false)

  useEffect(() => {
    if (!q) { setResults([]); return }
    const controller = new AbortController()
    const id = setTimeout(async () => {
      setLoading(true)
      try {
        const r = await onSearch(q)     // real impl would pass controller.signal
        setResults(r)
      } catch (e) {
        if ((e as Error).name !== 'AbortError') setResults([])
      } finally {
        setLoading(false)
      }
    }, 300)
    return () => { clearTimeout(id); controller.abort() }
  }, [q, onSearch])

  return (
    <div>
      <input value={q} onChange={(e) => setQ(e.target.value)} aria-label="Search" />
      {loading && <p>Loading…</p>}
      <ul>{results.map((r) => <li key={r.id}>{r.title}</li>)}</ul>
    </div>
  )
}

Infinite scroll with IntersectionObserver 
function useInfinite<T>(fetchPage: (page: number) => Promise<T[]>) {
  const [items, setItems] = useState<T[]>([])
  const [page, setPage] = useState(1)
  const [done, setDone] = useState(false)
  const sentinel = useRef<HTMLDivElement>(null)

  useEffect(() => {
    let cancelled = false
    fetchPage(page).then((newItems) => {
      if (cancelled) return
      if (newItems.length === 0) setDone(true)
      else setItems((prev) => [...prev, ...newItems])
    })
    return () => { cancelled = true }
  }, [page, fetchPage])

  useEffect(() => {
    if (done || !sentinel.current) return
    const io = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) setPage((p) => p + 1)
    })
    io.observe(sentinel.current)
    return () => io.disconnect()
  }, [done])

  return { items, sentinel, done }
}
Modal component with focus trap & portal

Tic-tac-toe with move history (Meta favorite)

Board = array of 9 squares. Store history of boards, not just current — enables time travel. Compute winner as a derived value each render, not stored. Immutable updates (slice instead of mutation). Toggle player from history length parity.

25Rapid-Fire Q&A

What is the difference between an element and a component?

A component is a function (or class) that returns UI. An element is the object returned by JSX / createElement that describes what to render: { type, props, key }. Elements are immutable and cheap; components are the recipe.

Why do keys matter?

They let React match list items across renders. Without stable keys, React matches by position and mistakes reorders for updates — corrupting state, focus, and animations.

useMemo vs React.memo?

useMemo memoizes a value inside a component. React.memo memoizes a component's render given its props. They solve different problems and are often used together.

What triggers a re-render?

Own state change, parent re-render (unless memoized + shallow-equal props), context change, external store change, or forced dispatch. Prop reference changes matter — objects/arrays created inline in the parent create new references each render.

What is Strict Mode?

A dev-only wrapper that intentionally double-invokes render, constructor, and effect setup/cleanup, and surfaces deprecation warnings. Its goal: shake out impure components and non-idempotent effects before concurrent features hit them in production. Does nothing in prod builds.

Difference between useEffect and useLayoutEffect?

useEffect runs asynchronously after paint. useLayoutEffect runs synchronously after DOM mutations but before paint. Use layout effect only when you must read the DOM and apply a compensating change without flashes.

Controlled vs uncontrolled components?

Controlled: parent owns value via value/onChange. Uncontrolled: DOM owns the value, read via a ref or on submit. Controlled is easier to validate; uncontrolled is cheaper for very large forms.

What does startTransition do?

Marks a state update as non-urgent. React can pause / interrupt / re-order it so urgent updates (typing) stay responsive. Returns [isPending, startTransition] from useTransition.

Why does React use synthetic events?

Cross-browser normalization, event pooling (until React 17 dropped it), and a single delegated listener at the root — cheaper than many DOM listeners for large trees. Since React 17 the listener is on the root container, not document.

What are portals?

A way to render children into a DOM node outside the parent's DOM hierarchy, while preserving React tree relationships (context, events). Used for modals, tooltips, toasts.

What is hydration?

The process of attaching React event listeners and state to server-rendered HTML. React walks the existing DOM instead of creating it, then makes it interactive. Hydration mismatches occur when the client's first render diverges from the server's HTML.

How does React batch state updates?

Multiple setState calls in the same tick are collected and processed in a single render. Before React 18, batching only happened in event handlers; React 18 batches in promises, timers, and native events as well.

What does flushSync do?

Forces the enclosed state update to render synchronously, escaping batching. Rare — use only when you must read the updated DOM before the next tick (e.g., before scrolling).

How does Suspense work under the hood?

Components can throw a promise. React catches it at the nearest Suspense boundary, marks that subtree as pending, renders the fallback, and attaches a continuation to retry when the promise resolves. In concurrent mode, unrelated work continues meanwhile.

Reconciliation heuristics in one sentence?

Assume same element type at same position = same instance; different type = destroy and remount; use keys to match list items across renders.
